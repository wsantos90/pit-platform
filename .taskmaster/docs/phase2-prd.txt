## 1. VISÃO GERAL DA FASE 2

### 1.1 O que é a Fase 2?
A Fase 1 do P.I.T (25 tasks) está concluída. O sistema é funcional mas o visual é inconsistente — seções puxadas de referências diferentes resultaram em fontes, espaçamentos, cores e tamanhos de componentes que não batem entre si. A Fase 2 consiste em:

1. **Refatoração do codebase** — cleanup, documentação, remoção de dead code, consolidação
2. **Brainstorming visual** — definir design system via UI/UX Pro Max skill, paleta de cores, tipografia
3. **Redesign visual** — página por página, componente por componente, usando skills do Claude Code
4. **Landing page pública** — nova página antes do login (campeonatos, rankings, resultados)
5. **SEO & Performance** — Open Graph, Lighthouse 90+, performance budget
6. **Testes de regressão visual** — component gallery, screenshot tests

### 1.2 Objetivos
- Zero hardcoded styles (tudo via tokens CSS)
- 100% das páginas migradas para design system
- 80%+ de componentes reutilizados (vs criados do zero)
- Lighthouse 90+ em landing page (performance + accessibility)
- WCAG AA compliance (contraste, keyboard nav, aria-labels)
- Toda página com estados: loading, empty, error, success, no-permission

### 1.3 Restrições
- **APENAS visual** — não mudar lógica de negócio, rotas, schema, tipos, API routes
- **Desktop-first** — responsividade mínima: 1440p / 1080p / 1366px
- **MVP-first** — lançar Bloco 1 (fundação + profile + auth) e iterar
- **Uma task = um branch = um PR** — facilitar rollback

### 1.4 Stack Existente
- Frontend: Next.js 16.1.6 (App Router, TypeScript) + Tailwind CSS 3.4.1
- Backend: Supabase (PostgreSQL 15+, Auth, RLS, Realtime, Storage)
- UI: shadcn/ui (11 componentes) + Lucide React + Recharts
- Deploy: Vercel (frontend) + VPS Ubuntu (cookie service)

### 1.5 Skills Disponíveis
- **UI/UX Pro Max** — gerar design system completo (MASTER.md)
- **frontend-design** — implementação production-grade de UI
- **stitch-brief / stitch-apply** — design briefs via Google Stitch
- **vercel-react-best-practices** — 62 regras de performance
- **simplify** — code review: reuso, qualidade, eficiência

### 1.6 Referências Visuais
- VirtualNexusArena.com — cards de torneio com banner, sidebar dark, transfers live, landing page pública
- 21st.dev — componentes UI de alta qualidade
- Dribbble — dashboards esports, cards, layouts de dados

## 9. DEFINITION OF DONE (POR TASK)

Uma task só está done quando:

1. `npm run build` — sem erros
2. `npm test` — testes passando
3. 100% alinhada com tokens CSS (zero hardcoded colors/spacing)
4. Usa componentes compartilhados quando aplicável
5. Tem estados: loading (skeleton), empty, error, success
6. Contraste WCAG AA verificado
7. Navegável via teclado (Tab, Enter, Escape)
8. Visual check em 1080p com dados reais
9. Visual check com dados vazios
10. Consistência de espaçamento/tipografia com páginas já redesenhadas
11. Copy revisada (labels, mensagens, CTAs em PT-BR)
12. Deploy preview na Vercel aprovado

## 11. RISCOS

| Risco | Mitigação |
|-------|-----------|
| Escopo crescer demais | MVP-first. Fase 5 é polish, não bloqueante |
| Misturar refatoração com mudança visual | Fase 0 separa refatoração do redesign |
| Criar componentes demais sem necessidade | Governança: reusar primeiro, justificar criação nova |
| Perder consistência visual | MASTER.md é SSOT. Toda task valida contra ele |
| Landing page virar projeto paralelo | Escopo mínimo: hero + campeonatos + rankings |
| Performance piorar | Budget (Task 46). Lighthouse em cada PR |
| Refatoração quebrar funcionalidade | Testes antes e depois. Deploy preview obrigatório |
